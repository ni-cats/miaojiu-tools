# 需求文档：图片 OCR 识别 / OCR 翻译

## 引言

本需求文档针对 ClipTool（macOS 剪贴板代码片段管理工具）新增**图片 OCR 文字识别**和 **OCR + 翻译**功能。

利用剪贴板已有的图片捕获能力（base64），基于**端侧开源 OCR 工具包**新增 OCR 文字识别和 OCR + 翻译功能，让用户可以快速从截图/图片中提取文字并翻译。OCR 识别完全在本地运行，无需网络，保护隐私。翻译阶段复用现有的混元大模型能力。

### 项目现状

- 剪贴板已支持图片读取（`clipboard.ts` 中 `readClipboard()` 返回 base64 图片，`writeClipboardImage()` 支持写入图片）
- 剪贴板监听模块（`clipboard-watcher.ts`）已支持图片变化检测，自动记录到历史
- 已接入腾讯混元大模型（`hunyuan.ts`），支持流式对话、AI 标题生成、AI 标签匹配
- 已有 AI 面板（`AiPanel.tsx`），包含对话窗口和预留的 AI 方案卡片（均为"即将推出"状态）
- 导航面板（`LauncherPanel.tsx`）已有图片 Base64 转换等内置工具
- IPC 通信架构成熟（main ↔ preload ↔ renderer），通过 `ipc.ts` 注册 handler，`preload/index.ts` 暴露 API
- 配置管理通过 `config.yaml` + `electron-store` 实现

### OCR 技术方案选型

采用**端侧开源 OCR 工具包**，不依赖大模型 API，具备以下优势：
- ✅ **离线可用**：无需网络连接，完全本地运行
- ✅ **隐私安全**：图片数据不出本机
- ✅ **零成本**：无 API 调用费用
- ✅ **低延迟**：本地推理，响应速度快

#### 方案对比

| 特性 | Tesseract.js | RapidOCR (ONNX Runtime) |
|------|-------------|------------------------|
| 语言 | 纯 JavaScript (WebAssembly) | Node.js 原生绑定 (onnxruntime-node) |
| 安装 | `npm install tesseract.js`，零原生依赖 | 需要 `onnxruntime-node` + ONNX 模型文件 |
| 中文识别精度 | 一般（Tesseract 引擎对中文支持有限） | **优秀**（基于 PaddleOCR PP-OCRv4 模型） |
| 英文识别精度 | 优秀 | 优秀 |
| 模型大小 | ~15MB（中英文语言包） | ~12MB（PP-OCRv4 轻量模型） |
| Electron 集成难度 | ⭐ 简单 | ⭐⭐ 中等 |
| 首次加载速度 | 较慢（需加载 WASM + 语言包） | 较快 |

#### 推荐方案

**首选 Tesseract.js**，理由：
1. 纯 JavaScript 实现，与 Electron 项目天然兼容，无需处理原生模块编译问题
2. 安装简单，`npm install tesseract.js` 即可
3. 支持 100+ 语言，满足多语言 OCR 需求
4. 社区活跃，文档完善，GitHub 35k+ stars
5. 支持 Worker 多线程，不阻塞主进程
6. 语言包可内置到应用中，实现完全离线

**如果后续中文识别精度不满足需求**，可考虑引入 RapidOCR 作为增强方案。

## 需求

### 需求 1：图片 OCR 文字识别（端侧）

**用户故事：** 作为一名开发者/办公用户，我希望能对剪贴板中的图片进行本地 OCR 文字识别，以便快速从截图中提取文字内容，避免手动输入，且无需联网。

#### 验收标准

1. WHEN 用户复制了一张图片到剪贴板 AND 在 ClipTool 中触发 OCR 识别 THEN 系统 SHALL 使用本地 Tesseract.js 引擎对图片进行文字识别，并将识别结果展示在界面上
2. WHEN OCR 识别完成 THEN 系统 SHALL 提供"复制识别结果"按钮，用户点击后将文字复制到剪贴板
3. WHEN 用户在剪贴板历史中选中一条图片类型的记录 THEN 系统 SHALL 提供 OCR 识别的快捷操作入口
4. IF 图片中无可识别文字 THEN 系统 SHALL 显示友好提示"未识别到文字内容"
5. WHEN OCR 引擎首次加载时 THEN 系统 SHALL 显示初始化进度（加载 WASM 引擎 + 语言包），后续使用缓存加速
6. WHEN OCR 识别过程中 THEN 系统 SHALL 显示加载状态，防止用户重复操作
7. WHEN 系统启动时 THEN 系统 SHALL 预加载 OCR 引擎和默认语言包（中英文），减少首次使用等待时间

#### 技术方案说明

- **OCR 引擎**：Tesseract.js（基于 WebAssembly 的纯 JavaScript OCR 引擎）
- **默认语言包**：`eng`（英文）+ `chi_sim`（简体中文），内置到应用资源中实现离线使用
- **运行位置**：在 Electron main 进程中通过 Tesseract.js Worker 运行，避免阻塞 UI
- **图片输入**：直接使用剪贴板读取的 base64 图片数据
- **IPC 通道**：新增 `ocr:recognize` IPC handler，接收 base64 图片，返回识别文字
- **UI 入口**：
  - 导航面板（LauncherPanel）新增"OCR 识别"工具卡片，与现有的"图片 Base64 转换"工具并列
  - 剪贴板历史面板（EditorPanel）中图片类型记录增加 OCR 操作按钮

---

### 需求 2：OCR + 翻译功能

**用户故事：** 作为一名需要阅读外文资料的用户，我希望能对图片中的文字进行本地 OCR 识别后自动翻译，以便快速理解外文截图内容。

#### 验收标准

1. WHEN 用户对图片触发"OCR 翻译"操作 THEN 系统 SHALL 先使用本地 Tesseract.js 进行 OCR 识别，再调用现有混元大模型将识别结果翻译为目标语言，并同时展示原文和译文
2. WHEN 翻译完成 THEN 系统 SHALL 提供"复制原文"和"复制译文"两个独立的复制按钮
3. IF 用户未设置目标翻译语言 THEN 系统 SHALL 默认翻译为中文（简体）
4. WHEN 用户在设置中 THEN 系统 SHALL 允许配置默认翻译目标语言（中文、英文、日文等常用语言）
5. WHEN OCR 翻译过程中 THEN 系统 SHALL 分步显示进度（"正在识别文字..." → "正在翻译..."）
6. IF 混元大模型翻译服务不可用（未配置密钥） THEN 系统 SHALL 仍然展示 OCR 识别的原文结果，并提示"翻译服务未配置，请在设置中配置混元大模型密钥"

#### 技术方案说明

- **OCR 阶段**：本地 Tesseract.js 引擎（离线），复用需求 1 的 OCR 能力
- **翻译阶段**：复用现有混元大模型 API（`hunyuan.ts`），通过 system prompt 指定翻译任务
- **流程**：OCR（本地离线） → 翻译（混元大模型，需网络） → 展示结果
- **IPC 通道**：新增 `ocr:translate` IPC handler，接收 base64 图片 + 目标语言，返回原文和译文
- **UI 入口**：
  - 导航面板（LauncherPanel）新增"OCR 翻译"工具卡片
  - 与 OCR 识别共享结果展示区域，翻译结果额外展示译文

## 边界情况与技术限制

1. **OCR 引擎初始化**：Tesseract.js 首次加载需要加载 WASM 引擎和语言包（约 15MB），应在应用启动时预加载，并将语言包内置到应用资源中
2. **中文识别精度**：Tesseract.js 对中文的识别精度不如 PaddleOCR，如果用户反馈中文识别效果不佳，后续可考虑引入 RapidOCR (ONNX Runtime) 作为增强
3. **图片质量**：低分辨率、模糊、倾斜的图片会影响 OCR 精度，可考虑在识别前进行简单的图片预处理（如灰度化、二值化）
4. **大图片处理**：超大图片可能导致内存占用过高，需要设置合理的图片大小上限或进行压缩
5. **离线 vs 在线**：OCR 识别完全离线可用；翻译功能依赖混元大模型网络连接，离线时仅展示 OCR 原文
6. **应用体积**：内置 Tesseract.js 语言包会增加约 15MB 应用体积，需要权衡
7. **并发识别**：同一时间只允许一个 OCR 任务运行，避免内存和 CPU 资源竞争
8. **语言包管理**：默认内置中英文语言包，后续可考虑支持用户按需下载其他语言包

## 成功标准

1. OCR 识别在中英文混合场景下可用，英文识别准确率 > 95%，中文识别准确率 > 85%
2. 从截图到获取识别文字的端到端时间 < 3 秒（引擎已预加载的情况下）
3. OCR 识别功能完全离线可用，无需任何网络连接
4. OCR 翻译功能在混元大模型已配置的情况下正常工作，翻译结果可读
5. 新功能不影响现有功能的正常使用
6. 应用启动后 OCR 引擎预加载完成时间 < 5 秒
7. UI 交互流畅，OCR 过程中有明确的加载状态提示
